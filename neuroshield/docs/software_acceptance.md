# Software acceptance: the software replay gate

This is the reproduction guide for the software-first acceptance gate (T19). It is written for
someone who did not write this code. Passing every step here is what allows hardware
integration (T20) to begin -- until then, the software replay demo is the product.

## Prerequisites

- `uv` installed (see the project README).
- A clean checkout of this repository (`neuroshield/`) with `data/external/` datasets **not**
  required for this gate -- it runs entirely on the committed replay fixture and the WESAD-trained
  model artifact already committed-adjacent in `artifacts/` (large binaries in
  `artifacts/models/*.joblib` are gitignored; if you don't have one, run
  `uv run python -m neuroshield.models.artifact` after downloading WESAD, or ask whoever ran T4-T9
  for the artifact).

## Steps

1. **Sync the environment.**
   ```bash
   uv sync
   uv run python -m neuroshield.smoke
   ```
   Expect: `software env ok` and every package version printed.

2. **Run the automated tests.**
   ```bash
   uv run pytest tests/ -q
   ```
   Expect: all tests pass (0 failures). This is the software equivalent of "the build is green."

3. **Run the acceptance script.** This does steps 4-9 below automatically and produces
   `artifacts/demo/software_acceptance.json`:
   ```bash
   uv run python scripts/software_acceptance.py
   ```

   Reading the script or running its steps by hand:

4. **Load the replay fixture.** The committed fixture is
   `data/fixtures/calm_motion_stress.ndjson` (generated by T10; regenerate with
   `uv run python -m neuroshield.runtime.synthetic_source --out data/fixtures/calm_motion_stress.ndjson --duration-sec 600 --seed 7 --session-id demo-001` if needed -- it is deterministic, so
   regenerating with the same seed produces byte-identical output).

5. **Start the backend in replay mode:**
   ```bash
   uv run python -m neuroshield.api.main
   ```
   This binds to `127.0.0.1:8000` by default. In another terminal:
   ```bash
   curl -X POST http://127.0.0.1:8000/api/v1/session/start \
     -H "Content-Type: application/json" \
     -d '{"source_mode":"replay","replay_path":"data/fixtures/calm_motion_stress.ndjson","session_id":"demo-001"}'
   curl -X POST http://127.0.0.1:8000/api/v1/calibration/start \
     -H "Content-Type: application/json" -d '{"quiet_seconds":150}'
   ```

6. **Start the dashboard:**
   ```bash
   cd web && npm install && npm run dev   # http://localhost:3000
   ```
   Open the printed local URL. You should see the connection state, current status badge,
   quality row, reasons, latest values, and a history chart, matching T17.

7. **Verify the state sequence.** Using `GET /api/v1/history` (or the dashboard's history table),
   confirm the session passes through, in order: `waiting`/`calibrating` (before calibration),
   `green` (quiet segment), `motion_paused` (motion burst), back to `green` (recovery),
   `amber`/`red` (the mild-stress-rise segment), and `poor_signal` (the sensor-fault segment).

8. **Verify backend restart behavior.** Stop the backend process (Ctrl-C), restart it, and call
   `GET /api/v1/status/latest` again *without* starting a new session. Expect `state: "waiting"`
   -- never a leftover `green` from before the restart. Then start a new session/calibration and
   confirm the dashboard reconnects and updates (refresh or use "Start session" again).

9. **Verify stale detection.** This is checked directly against the state machine in the
   automated test suite (`tests/test_status.py::TestStaleDetection`) and exercised again by
   `scripts/software_acceptance.py`, since a 120-second real-time gap is impractical to demonstrate
   by hand in a document.

10. **Evidence.** `scripts/software_acceptance.py` records the exact commands run, package
    versions, the git commit, the files it produced, and a pass/fail note for each check into
    `artifacts/demo/software_acceptance.json`. There is no screen recording or screenshot in this
    environment (no browser automation tool was available when this gate was last run) -- the
    JSON transcript and the passing automated test suite are the evidence of record instead.

## What passing this gate means

Once every check in `artifacts/demo/software_acceptance.json` is `"pass"`, the software-first MVP
is ready and hardware integration (T20) may begin. Hardware only needs to replace the event
*source* -- everything from features onward must not need to change.
